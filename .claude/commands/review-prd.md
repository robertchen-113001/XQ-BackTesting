你的任務是以資深 PM（產品經理）的角度，審查指定的 PRD 文件。請嚴格遵守以下 SOP：

1. **確認審查目標**：若使用者未指定 PRD 文件，請詢問要審查哪一份（通常在 `docs/prd/` 下）。

2. **前置閱讀**：
   - 讀取 `memory/core.md`，確認專案願景與最高鐵律
   - 讀取 `memory/ltm.md`，確認術語定義與現有架構決策
   - 讀取目標 PRD 文件的完整內容

3. **以 RD 視角審查以下面向**：
   - **技術可行性**：需求描述是否清晰到足以實作？是否有技術上的模糊地帶？
   - **邊界條件**：是否有未定義的 Edge Case？錯誤處理是否完整？
   - **前後端分工**：哪些是前端邏輯？哪些需要 API 支援？介面是否定義清楚？
   - **效能考量**：是否有潛在的效能瓶頸（如大量 AI 呼叫、DOM 更新等）？
   - **與現有架構的衝突**：需求是否與 `docs/prd/` 其他規格或 `frontend/` 現有實作相衝突？
   - **缺漏需求**：是否有明顯遺漏的使用者情境或功能分支？

4. **輸出審查結果**：
   - 將發現的問題以 `- [ ] [YYYY-MM-DD] 問題描述 → 相關 PRD §章節` 格式補充至 `docs/open-questions.md` 的「未解決問題」區塊頂端（日期遞減，最新在上）
   - 合併同質性高的問題（描述本質相同、只是來源 REQ 不同者，整合為一條）
   - 若問題已解決，移入「已解決問題」區塊，加上「已於 YYYY-MM-DD 修正」說明
   - `docs/open-questions.md` 維持兩區塊格式：「## 未解決問題（按日期遞減）」與「## 已解決問題（按日期遞減）」
   - 向使用者摘要審查結論（3–5 點最關鍵的發現）

5. **不要**主動修改 PRD 文件本身，除非使用者明確要求。
