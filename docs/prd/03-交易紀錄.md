## 2. 統一 CSV 欄位規格

本節為整份系統的**唯一 CSV 格式參考來源**。

- **上傳（匯入）**：使用者只需提供「必要欄位」；計算欄位由 server 統一重算，上傳時即使填入也會被覆蓋
- **匯出**：server 輸出完整 19 欄，格式與此節定義相同

### 2.1 完整欄位定義（19 欄）

| # | 欄位 ID | 欄位名稱 | 格式 / 說明 | 必要（上傳）| 來源 |
|---|--------|---------|------------|:----------:|------|
| 1 | `Name` | 商品名稱 | string | — | 系統依商品代碼查詢 |
| 2 | `Code` | 商品代碼 | XQ 商品代碼（如 `2330.TW`）| ✓ | 使用者提供 |
| 3 | `Number` | 序號 | integer，同一商品遞增 | — | 系統自動產生 |
| 4 | `EntryDate` | 進場時間 | 見 §2.4 | ✓ | 使用者提供 |
| 5 | `EntryDirection` | 進場方向 | `買進`、`B`、`賣出`、`S`；見 §2.6 | — | 預設 `買進` |
| 6 | `EntryPrice` | 進場價格 | double | ✓ | 使用者提供 |
| 7 | `ExitDate` | 出場時間 | 見 §2.4；未出場填 `--` | ✓ | 使用者提供 |
| 8 | `ExitDirection` | 出場方向 | 系統自動推算（見 §2.6）；未出場填 `--` | — | 系統自動推算 |
| 9 | `ExitPrice` | 出場價格 | double；未出場填 `--`（見 §2.5）| ✓ | 使用者提供 |
| 10 | `HoldingPeriod` | 持有區間 | integer（交易日數）| — | server 計算 |
| 11 | `NumberOfTrades` | 交易數量 | integer；`-1` 由 server 依設定換算 | ✓ | 使用者提供 |
| 12 | `ProfitLoss` | 獲利金額 | double，ROUNDDOWN 至整數 | — | server 計算 |
| 13 | `ReturnRate` | 報酬率 | double，保留 6 位小數 | — | server 計算 |
| 14 | `NetProfit_Amount` | 累計獲利金額 | double，ROUNDDOWN 至整數 | — | server 計算 |
| 15 | `NetProfit` | 累計報酬率 | double，保留 6 位小數 | — | server 計算 |
| 16 | `PeriodHigh` | 區間最高價 | double，持有區間內最高成交價 | — | server 計算 |
| 17 | `PeriodLow` | 區間最低價 | double，持有區間內最低成交價 | — | server 計算 |
| 18 | `EntrySignal` | 進場訊息 | string，策略觸發條件說明 | — | 使用者提供或策略產生 |
| 19 | `ExitSignal` | 出場訊息 | string，出場條件說明（如 `停利 8 %`）| — | 使用者提供或策略產生 |

### 2.2 必要欄位 vs. 計算欄位

**必要欄位（上傳最低需求，共 6 欄）：**
`Code`、`EntryDate`、`EntryPrice`、`ExitDate`、`ExitPrice`、`NumberOfTrades`

**計算欄位（server 統一重算；上傳時若填入亦被覆蓋）：**
`Name`、`Number`、`HoldingPeriod`、`ProfitLoss`、`ReturnRate`、`NetProfit_Amount`、`NetProfit`、`PeriodHigh`、`PeriodLow`

**選填欄位（使用者可提供，有值則保留）：**
`EntryDirection`、`ExitDirection`（系統仍以 §2.6 規則重算）、`EntrySignal`、`ExitSignal`

### 2.4 時間格式規範

| 情境 | 格式 | 範例 |
|------|------|------|
| 日頻（選股中心、策略雷達）| `yyyy/MM/dd` | `2026/04/02` |
| 分鐘頻（自動交易）| `yyyy/MM/dd HH:mm:ss` | `2026/04/01 09:00:00` |
| 未出場（持倉中）| `--` | `--` |

> ⚠️ 舊版 Spec 記載的 `yyyy:MM:dd`（冒號分隔）為筆誤，正確分隔符號為斜線 `/`。

### 2.5 未出場欄位佔位規則

當交易尚未出場時，出場相關欄位填入 `--`（**三平台均適用**，不限自動交易）。

| 欄位 | 佔位值 | 系統處理 |
|------|--------|---------|
| 出場時間 | `--` | 以回測結束日填入 |
| 出場方向 | `--` | 依進場方向反推 |
| 出場價格 | `--` | 以回測結束日收盤價填入 |
| 獲利金額 | `--` | server 計算後填入 |
| 報酬率 | `--` | server 計算後填入 |

持有區間、累計欄位、區間高低價依當前最新資料計算後填入，不用 `--`。

**部分欄位不一致的處理（僅上傳情境）：**

若上傳的 CSV 中，{出場時間, 出場價格} 其中一欄有值、另一欄為 `--`，系統彈出警示框讓使用者確認：

```
+------------------------------------------------------------+
|  ⚠️ 出場資料不一致                                         |
|------------------------------------------------------------|
|  以下資料列的出場時間與出場價格僅填入其中一個：            |
|  ・第 N 筆（2330.TW，進場 2026/01/05）                     |
|                                                            |
|  [使用回測結束日收盤價補齊缺漏欄位]  [中止上傳]           |
+------------------------------------------------------------+
```

### 2.6 進出場方向規則

| 規則 | 說明 |
|------|------|
| 未填入進場方向 | 系統預設 `買進`，出場方向自動推算為 `賣出` |
| 支援縮寫 | `B` = `買進`；`S` = `賣出` |
| 出場方向 | 固定為進場方向的反向（`買進` → `賣出`；`賣出` → `買進`）；系統自動推算，上傳時填入的值會被覆蓋 |
| 未出場 | 出場方向填 `--` |

## 3. 交易數量規則

交易數量決定每筆進場時買入多少股（或口），影響獲利金額、報酬率等所有金額類欄位的計算結果。

| 模式 | 說明 | 固定值 | 適用來源 |
|------|------|--------|---------|
| **腳本** | 由交易腳本內自行指定買賣數量，server 直接使用，不做換算 | 依腳本邏輯 | 僅自動交易 |
| **等額** | 每筆進場使用固定 10 萬元換算股數（= ROUNDDOWN(100,000 ÷ 進場價格 ÷ Lots)）；期貨固定 1 口 | 固定 10 萬元 | 全平台 |
| **等量** | 每筆進場固定 1 張（= 1,000 股 ÷ Lots）；期貨固定 1 口 | 固定 1 張 | 全平台 |
| **等比** | 依「最大同時持有筆數」比例分配，每筆佔回測部位 1/N；無固定進場金額，強制搭配 TWRR | — | 全平台 |
| **依紀錄** | 直接採用 CSV 中的 `NumberOfTrades` 欄位值；若任一資料列為 `-1`，server 以**等量**補算並標記警告 | 依欄位值 | 僅使用者匯入 |

**特殊值 `-1` 的處理：**

- 各平台 XS 產生的交易紀錄（自動交易腳本自訂除外）統一輸出 `NumberOfTrades = -1`
- 量化積木輸出 `-1` 時，server 以**等量**模式換算
- 因子投資輸出 `-1` 時，server 以**等比**模式換算
- LocalBella 輸出 `-1` 時，server 以**等量**模式換算
- **使用者上傳（等額 / 等量 / 等比 模式）**：全部資料列依選擇的模式重新換算，忽略 CSV 中原有數值
- **使用者上傳（依紀錄模式）**：採用 CSV 中的原始數值；若某列為 `-1`，以**等量**補算並在報告中標記

**等比模式限制：**

等比模式無固定進場金額，無法計算 MIR、MWRR、MOIC 等金額類算法。選擇等比模式時，報酬率算法強制鎖定「時間加權報酬率（TWRR）」，其他選項不可選取。

> ℹ️ **User Story**
> 作為報表統計服務，我需要接收統一格式的交易數量（預設 -1），並根據平台設定自動轉換為腳本 / 等額 / 等量 / 等比等模式，確保各平台的計算邏輯一致。

## 4. CSV 匯入規格

### 4.1 上傳流程

1. 使用者於 XScript 編輯器選單「**上傳交易紀錄...**」開啟「上傳回測設定對話框」（§4.4）
2. 使用者在對話框內點擊「Browse...」選取 `.csv` 檔案
3. 系統執行格式驗證（§4.2），驗證失敗顯示錯誤清單（§4.3），通過後啟用「開始計算」
4. 使用者確認設定後點擊「開始計算」，server 依設定重算所有計算欄位並產出報告

> 📐 **前端 Prototype — 選單入口**
>
> ```text
> [XScript 編輯器 選單]
> ┌─────────────────────┐
> │ 檔案(F)             │
> ├─────────────────────┤
> │ 新增(N)    Ctrl+N   │
> │ 開啟(O)    Ctrl+O   │
> │ 開啟XS檔案...       │
> │ 關閉(C)        F4   │
> │ ─────────────────── │
> │ 儲存(S)    Ctrl+S   │
> │ 儲存XS檔案...       │
> │ 另存新檔(A)...      │
> │ 全部儲存(L)         │
> │ ─────────────────── │
> │ 開啟回測報告...     │  ← .BTReport / .BTReportNew
> │ 上傳交易紀錄...     │  ← .csv
> │ ─────────────────── │
> │ 匯出               │
> │ 匯入               │
> └─────────────────────┘
> ```

### 4.2 欄位驗證規則

| 欄位 | 驗證規則 |
|------|---------|
| 商品代碼 | 非空字串 |
| 進場時間 | 符合 `yyyy/MM/dd` 或 `yyyy/MM/dd HH:mm:ss` |
| 進場價格 | 可解析為 double，且 > 0 |
| 出場時間 | 符合時間格式，或為 `--` |
| 出場價格 | 可解析為 double 且 > 0，或為 `--`；出場時間非 `--` 時不得為 `--`（不一致時見 §2.5）|
| 交易數量 | 可解析為 integer；`-1` 為合法值 |
| 進場方向 | 若存在，需為 `買進`、`B`、`賣出`、`S` |
| 時間先後 | 出場時間非 `--` 時，出場時間需 ≥ 進場時間 |
| 方向一致性 | 若同時提供進場方向與出場方向，兩者需互為反向（`買進`/`B` ↔ `賣出`/`S`）|

### 4.3 錯誤訊息規格

| 錯誤類型 | 顯示訊息 |
|---------|---------|
| 必要欄位缺失 | 「請確認交易紀錄是否包含以下欄位：商品代碼、進場時間、進場價格、出場時間、出場價格、交易數量。」 |
| 欄位重複 | 「請確認交易紀錄是否有重複的欄位。」 |
| 格式錯誤 | 「請確認 [欄位名稱] 的格式。」 |
| 時間順序錯誤 | 「第 [行號] 筆：出場時間不得早於進場時間。」 |
| 方向相同 | 「第 [行號] 筆：進場方向與出場方向不得相同，應互為反向。」 |
| 出場資料不一致 | 彈出確認框（見 §2.5），讓使用者選擇補齊或中止 |

> 📐 **前端 Prototype — 錯誤清單**
> - 格式錯誤清單嵌入於「上傳回測設定對話框」內（§4.4 Prototype 最下方區塊）
> - 有錯誤時展開並禁用「開始計算」；無錯誤時收起
> - 使用者修正 CSV 後重新點「Browse...」即可重新驗證

> ℹ️ **User Story**
> 作為使用者，當我上傳格式錯誤的 CSV 時，
> 我希望能看到明確的錯誤說明，告訴我哪個欄位出了什麼問題，
> 以便快速修正並重新匯入。

### 4.4 上傳回測設定對話框

使用者上傳 CSV 後需設定計算參數。包含三個模組（交易設定、交易費用、槓桿），省略平台差異模組。

**欄位規格：**

| 模組 | 欄位 | 預設值 | 說明 |
|------|------|--------|------|
| 交易設定 | 交易數量 | 依紀錄 | 依紀錄 / 等額 / 等量 / 等比（「腳本」選項不顯示）|
| 交易設定 | 報酬率算法 | 時間加權報酬率 | 等比時強制鎖定 TWRR |
| 交易費用 | 股票手續費 | 0.1425 % | — |
| 交易費用 | 股票交易稅 | 0.3 % | — |
| 交易費用 | 期貨手續費 | 100 元/口 | — |
| 交易費用 | 期貨交易稅 | 0.3 % | — |
| 槓桿 | 期貨保證金成數 | 13.5 % | 勾選啟用 |

**回測日期驗證：**

- 回測起始日與結束日為**必填**，系統依此計算績效曲線
- 驗證所有交易的進出場時間是否落在日期範圍內，若有超出則彈出警示框：

## 6. BTReportNew

### 6.1 入口

BTReportNew 與 CSV 上傳共用 XScript 編輯器選單同一區塊（見 §4.1 選單 Prototype）。系統依副檔名自動判斷版本：

| 副檔名 | 版本 | 開啟方式 |
|--------|------|---------|
| `.BTReport` | 舊版 | 以**舊版 UI** 開啟（功能受限）|
| `.BTReportNew` | 新版 | 以**新版 Web UI** 直接開啟，支援所有報告功能 |

### 6.2 舊版 BTReport 相容

- 以舊版 UI 開啟，僅支援舊版功能
- 使用者點選「**重新回測**」時，系統彈出新版**執行回測對話框**（PRD 04 §3），以新版參數設定重新執行，產出 `.BTReportNew`

### 6.3 BTReportNew 匯入

- 匯入後**不需要重新執行**報表統計即可直接使用所有報告功能
- 匯入時可自訂回測報告標題（預設使用檔案內儲存的標題）

> ℹ️ **User Story**
> 作為使用者，我希望匯入新版 BTReportNew 後可以直接使用所有報告功能，
