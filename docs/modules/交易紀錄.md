## 2. 統一 CSV 欄位規格

本節為整份系統的**唯一 CSV 格式參考來源**。

- **上傳（匯入）**：使用者只需提供「必要欄位」；計算欄位由 server 統一重算，上傳時即使填入也會被覆蓋
- **匯出**：server 輸出完整 19 欄，格式與此節定義相同

### 2.1 完整欄位定義（19 欄）

| # | 欄位 ID | 欄位名稱 | 格式 / 說明 | 必要（上傳）| 來源 |
|---|--------|---------|------------|:----------:|------|
| 1 | `Name` | 商品名稱 | string | — | 系統依商品代碼查詢 |
| 2 | `Code` | 商品代碼 | XQ 商品代碼（如 `2330.TW`）| ✓ | 使用者提供 |
| 3 | `Number` | 序號 | integer，同一商品遞增 | — | 系統自動產生 |
| 4 | `EntryDate` | 進場時間 | 見 §2.4 | ✓ | 使用者提供 |
| 5 | `EntryDirection` | 進場方向 | `買進`、`B`、`賣出`、`S`；見 §2.6 | — | 預設 `買進` |
| 6 | `EntryPrice` | 進場價格 | double | ✓ | 使用者提供 |
| 7 | `ExitDate` | 出場時間 | 見 §2.4；未出場填 `--` | ✓ | 使用者提供 |
| 8 | `ExitDirection` | 出場方向 | 系統自動推算（見 §2.6）；未出場填 `--` | — | 系統自動推算 |
| 9 | `ExitPrice` | 出場價格 | double；未出場填 `--`（見 §2.5）| ✓ | 使用者提供 |
| 10 | `HoldingPeriod` | 持有區間 | integer（交易日數）| — | server 計算 |
| 11 | `NumberOfTrades` | 交易數量 | integer；XS 固定輸出 `-1`，server 依設定換算 | ✓ | 使用者提供 / server 換算 |
| 12 | `ProfitLoss` | 獲利金額 | double，ROUNDDOWN 至整數 | — | server 計算 |
| 13 | `ReturnRate` | 報酬率 | double，保留 6 位小數 | — | server 計算 |
| 14 | `NetProfit_Amount` | 累計獲利金額 | double，ROUNDDOWN 至整數 | — | server 計算 |
| 15 | `NetProfit` | 累積報酬率 | double，保留 6 位小數 | — | server 計算 |
| 16 | `PeriodHigh` | 區間最高價 | double，持有區間內最高成交價 | — | server 計算 |
| 17 | `PeriodLow` | 區間最低價 | double，持有區間內最低成交價 | — | server 計算 |
| 18 | `EntrySignal` | 進場訊息 | string，策略觸發條件說明 | — | 使用者提供或策略產生 |
| 19 | `ExitSignal` | 出場訊息 | string，出場條件說明（如 `停利 8 %`）| — | 使用者提供或策略產生 |

### 2.2 必要欄位 vs. 計算欄位

**必要欄位（上傳最低需求，共 6 欄）：**
`Code`、`EntryDate`、`EntryPrice`、`ExitDate`、`ExitPrice`、`NumberOfTrades`

**計算欄位（server 統一重算；上傳時若填入亦被覆蓋）：**
`Name`、`Number`、`HoldingPeriod`、`ProfitLoss`、`ReturnRate`、`NetProfit_Amount`、`NetProfit`、`PeriodHigh`、`PeriodLow`

**選填欄位（使用者可提供，有值則保留）：**
`EntryDirection`、`ExitDirection`（系統仍以 §2.6 規則重算）、`EntrySignal`、`ExitSignal`

> 此表說明各平台現行輸出如何對應至公版欄位。三平台日後輸出**統一依公版 19 欄**，不再維護各自格式。

| # | 公版欄位 | 選股中心 | 策略雷達 | 自動交易 |
|---|---------|:-------:|:-------:|:-------:|
| 1 | 商品名稱 | ✓ | ✓ | ✓ |
| 2 | 商品代碼 | ✓ | ✓ | ✓ |
| 3 | 序號 | ✓ | ✓ | ✓ |
| 4 | 進場時間 | ✓（日期）| ✓（日期）| ✓（含時分秒）|
| 5 | 進場方向 | ✓ | ✓ | ✓ |
| 6 | 進場價格 | ✓ | ✓ | ✓ |
| 7 | 出場時間 | ✓（日期）| ✓（日期）| ✓（含時分秒 / `--`）|
| 8 | 出場方向 | ✓ | ✓ | ✓ 或 `--` |
| 9 | 出場價格 | ✓ | ✓ | ✓ 或 `--` |
| 10 | 持有區間 | ✓ | ✓ | ✓ |
| 11 | 交易數量 | 輸入 `-1`，server 後製 | 輸入 `-1`，server 後製 | server 後製 / 腳本自訂 |
| 12 | 獲利金額 | —（server 計算）| —（server 計算）| —（server 計算）|
| 13 | 報酬率 | —（server 計算）| —（server 計算）| —（server 計算）|
| 14 | 累計獲利金額 | —（server 計算）| —（server 計算）| —（server 計算）|
| 15 | 累計報酬率 | —（server 計算）| —（server 計算）| —（server 計算）|
| 16 | 區間最高價 | —（server 計算）| —（server 計算）| —（server 計算）|
| 17 | 區間最低價 | —（server 計算）| —（server 計算）| —（server 計算）|
| 18 | 進場訊息 | —（留空）| ✓ | ✓ |
| 19 | 出場訊息 | ✓（原「訊息」欄）| ✓ | ✓ |

> **選股中心訊息欄位說明**：選股中心原始格式僅有單欄「訊息」，語意為出場條件（如 `停利 8 %`），統一對應至第 19 欄「出場訊息」；第 18 欄「進場訊息」留空。

### 2.4 時間格式規範

兩種格式均合法，server 依欄位精度決定處理方式：

| 情境 | 格式 | 範例 |
|------|------|------|
| 日頻（選股中心、策略雷達）| `yyyy/MM/dd` | `2026/04/02` |
| 分鐘頻（自動交易）| `yyyy/MM/dd HH:mm:ss` | `2026/04/01 09:00:00` |
| 未出場（持倉中）| `--` | `--`（僅 XS 產生，使用者上傳不支援）|

### 2.5 未出場欄位佔位規則

當交易尚未出場時，出場相關欄位填入 `--`（**僅 XS 平台產生的資料適用**；使用者上傳不支援 `--`，見 §4）。

| 欄位 | 佔位值 | 系統處理 |
|------|--------|---------|
| 出場時間 | `--` | 以回測結束日填入 |
| 出場方向 | `--` | 依進場方向反推 |
| 出場價格 | `--` | 以回測結束日收盤價填入 |

**出場資料不一致（僅上傳情境）：**

若上傳的 CSV 中，出場時間與出場價格僅填入其中一欄，視為格式錯誤，直接顯示於錯誤清單，請使用者修正後重新上傳。

### 2.6 進出場方向規則

| 規則 | 說明 |
|------|------|
| 未填入進場方向 | 系統預設 `買進`，出場方向自動推算為 `賣出` |
| 支援縮寫 | `B` = `買進`；`S` = `賣出` |
| 出場方向 | 固定為進場方向的反向（`買進` → `賣出`；`賣出` → `買進`）；屬系統派生欄位，使用者上傳時填入的值一律被覆蓋，不做方向一致性驗證 |
| 未出場 | 出場方向填 `--` |

## 3. 交易數量規則

交易數量決定每筆進場時買入多少股（或口），影響獲利金額、報酬率等所有金額類欄位的計算結果。

| 模式 | 說明 | 固定值 | 適用來源 |
|------|------|--------|---------|
| **腳本** | 由交易腳本內自行指定買賣數量，server 直接使用，不做換算 | 依腳本邏輯 | 僅自動交易 |
| **等額** | 每筆進場使用固定 10 萬元換算：股票 = ROUNDDOWN(100,000 ÷ 進場價格 ÷ Lots)，期貨同公式換算口數（= ROUNDDOWN(100,000 ÷ 合約報價 ÷ Lots)）；結果取最大可買入整數股數 / 口數 | 固定 10 萬元 | 全平台 |
| **等量** | 每筆進場固定 1 張（= 1,000 股 ÷ Lots）；期貨固定 1 口 | 固定 1 張 / 1 口 | 全平台 |
| **等比** | 依「最大同時持有筆數」比例分配，每筆佔回測部位 1/N；無固定進場金額，強制搭配時間加權報酬率 | — | 全平台 |
| **依紀錄** | 直接採用 CSV 中的 `NumberOfTrades` 欄位值；若 CSV 含 `-1`，不允許選取此模式（選擇時即時提示格式錯誤）| 依欄位值 | 僅使用者匯入 |

**特殊值 `-1` 的處理：**

- 選股中心、策略雷達固定輸出 `-1`，server 依使用者選定的交易數量模式換算
- 自動交易（腳本自訂）輸出實際數量，server 直接使用
- 使用者上傳（等額 / 等量 / 等比模式）：全部資料列依選定模式重算，忽略 CSV 原有數值
- 使用者上傳（依紀錄模式）：CSV 含 `-1` 時不允許選取，需改選其他模式

**等比模式限制：**

等比模式無固定進場金額，無法計算金額欄位。選擇等比模式時，報酬率算法強制鎖定「時間加權報酬率」，其他選項不可選取。

## 4. CSV 匯入規格

### 4.1 上傳流程

1. 使用者於 XScript 編輯器選單「**上傳交易紀錄...**」開啟「上傳回測設定對話框」（§4.4）
2. 使用者在對話框內點擊「Browse...」選取 `.csv` 檔案
3. 系統執行格式驗證（§4.2），驗證失敗顯示錯誤清單（§4.3），通過後啟用「開始計算」
4. 使用者確認設定後點擊「開始計算」，server 依設定重算所有計算欄位並產出報告

> 回測日期自動取交易紀錄中**最早進場日**至**最晚出場日**，不需使用者填寫。\
> 使用者上傳的 CSV 不支援 `--`（持倉中）欄位；所有交易必須有完整出場時間與出場價格。

### 4.2 欄位驗證規則

| 欄位 | 驗證規則 |
|------|---------|
| 商品代碼 | 非空字串 |
| 進場時間 | 符合 `yyyy/MM/dd` 或 `yyyy/MM/dd HH:mm:ss` |
| 進場價格 | 可解析為 double，且 > 0 |
| 出場時間 | 符合時間格式（不接受 `--`；使用者上傳不支援持倉中交易）|
| 出場價格 | 可解析為 double，且 > 0；不得為空或 `--` |
| 出場資料完整性 | 出場時間與出場價格須同時有值，任一缺漏視為格式錯誤 |
| 交易數量 | 可解析為 integer；`-1` 為合法值 |
| 進場方向 | 若存在，需為 `買進`、`B`、`賣出`、`S` |
| 時間先後 | 出場時間非 `--` 時，出場時間需 ≥ 進場時間 |

### 4.3 錯誤訊息規格

| 錯誤類型 | 顯示訊息 |
|---------|---------|
| 必要欄位缺失 | 「缺少必要欄位：[欄位1]、[欄位2]，請確認後重新上傳。」 |
| 欄位重複 | 「欄位重複：[欄位名稱]，請確認後重新上傳。」 |
| 格式錯誤 | 「請確認 [欄位名稱] 的格式。」 |
| 時間順序錯誤 | 「第 [行號] 筆：出場時間不得早於進場時間。」 |
| 出場資料不一致 | 「第 [行號] 筆：出場時間與出場價格僅填入其中一欄，請修正後重新上傳。」 |
| 依紀錄模式格式錯誤 | 「CSV 中含有 NumberOfTrades = -1，無法使用「依紀錄」模式。請改選其他交易數量模式。」 |

### 4.4 上傳回測設定對話框

使用者上傳 CSV 後需設定計算參數。包含三個模組（交易設定、交易費用、槓桿），省略平台差異模組。

**欄位規格：**

| 模組 | 欄位 | 預設值 | 說明 |
|------|------|--------|------|
| 交易設定 | 交易數量 | 依紀錄 | 依紀錄 / 等額 / 等量 / 等比（「腳本」選項不顯示；CSV 含 -1 時不可選依紀錄）|
| 交易設定 | 報酬率算法 | 時間加權報酬率 | 等比時強制鎖定 |
| 交易費用 | 股票手續費 | 0.1425 % | — |
| 交易費用 | 股票交易稅 | 0.3 % | — |
| 交易費用 | 期貨手續費 | 100 元/口 | — |
| 交易費用 | 期貨交易稅 | 0.3 % | — |
| 槓桿 | 期貨保證金成數 | 13.5 % | 勾選啟用 |

> 回測日期由 server 自動取交易紀錄中最早進場日至最晚出場日，對話框顯示偵測結果（唯讀），不可手動調整。

## 5. CSV 匯出規格

### 5.1 匯出內容

使用者可於回測報告 UI 匯出本次回測的完整交易紀錄 CSV。

**資料範圍：**

- 完整 19 欄，欄位順序與 §2.1 相同
- 第一列為標題列，使用 §2.1 的中文欄位名稱（如「進場時間」、「報酬率」…），不使用英文欄位 ID
- 所有計算欄位均為 server 計算後的完整數值，**不含佔位值 `--`**

**持倉中交易的處理：**

回測期末仍未出場的交易，server 在計算階段即強制以回測結束日作為出場時間，以期末收盤價作為出場價格，並反推出場方向。因此匯出的 CSV 中，所有交易均具備完整的出場欄位數值，不會出現 `--`。

| 欄位 | 匯出值 |
|------|--------|
| 出場時間 | 回測結束日（`yyyy/MM/dd` 或含時分秒）|
| 出場方向 | 進場方向的反向（`買進` → `賣出`；`賣出` → `買進`）|
| 出場價格 | 回測結束日收盤價 |

> `--` 佔位規則（§2.5）僅適用於 XS 平台輸出的**原始資料**，進入 server 計算後即被替換為實際數值，匯出 CSV 不沿用 `--`。

### 5.2 檔名命名規則

| 情境 | 格式 | 範例 |
|------|------|------|
| 一般匯出 | `策略名稱_交易數量方式.csv` | `DIF-MACD從負翻正_等比.csv` |
| 使用者匯入後再匯出 | `原檔名_export.csv` | `my_trades_export.csv` |

## 6. BTReportNew

### 6.1 入口

BTReport 與 CSV 上傳同屬 XScript 編輯器選單同一區塊（見 §4.1）。兩者功能不同：

| 選單項目 | 行為 |
|---------|------|
| 開啟回測報告... | 直接 Browse 選取 `.BTReport` / `.BTReportNew`，無對話框，讀取已存好的報告資料直接開啟 |
| 上傳交易紀錄... | 開啟「上傳回測設定對話框」（§4.4），需設定計算參數後由 server 重算 |

系統依副檔名判斷開啟方式：

| 副檔名 | 版本 | 開啟方式 |
|--------|------|---------|
| `.BTReport` | 舊版 | 以**舊版 UI** 開啟（功能受限）|
| `.BTReportNew` | 新版 | 以**新版 Web UI** 直接開啟，支援所有報告功能 |

### 6.2 舊版 BTReport 相容

- 以舊版 UI 開啟，僅支援舊版功能
- 使用者點選「**重新回測**」時，系統彈出新版**執行回測對話框**（PRD 04 §3），以新版參數設定重新執行
- 執行完成後產出**回測報表**，使用者可另行匯出為 `.BTReportNew`

### 6.3 BTReportNew 匯入

- 匯入後**不需要重新執行**報表統計即可直接使用所有報告功能

### 6.4 BTReportNew 匯出

- 格式：`.BTReportNew`
- 命名：依回測報告標題命名
- 包含完整回測結果（設定、統計、交易明細），可於 Web UI 重新開啟
