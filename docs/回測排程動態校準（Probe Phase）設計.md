## **問題描述** {#問題描述}

目前 timeout 主要發生在**策略計算階段**（2 分鐘上限）。對於計算量大的策略，server 以固定並行數排程，導致部分商品計算來不及完成而 timeout。

---

## **提案：在正式排程前加入校準階段** {#提案：在正式排程前加入校準階段}

### **核心概念** {#核心概念}

先跑少量探針商品，取得實測耗時，讓 server 動態決定本次回測的 per-machine concurrency，避免 timeout。

---

### **修改範圍（最小）** {#修改範圍（最小）}

#### Step 1：Probe Phase {#step-1：probe-phase}

- 從整批商品中，抽取少量代表性商品（建議 **10 檔**，取成交量前段熱門股）  
- 優先排程這批探針商品  
- 收集每檔的實際 `elapsed_sec`（API 已有此欄位）

#### Step 2：Concurrency 估算 {#step-2：concurrency-估算}

```
p95_time = 探針中完成商品的 P95 耗時（排除 timeout 商品，避免污染估計）
safe_concurrency = floor(120s / p95_time)
```

#### Step 3：套用至後續排程 {#step-3：套用至後續排程}

- 剩下商品排程時，每台機器一次最多派 `safe_concurrency` 檔  
- **VIP 權重、機器空閒度、核心分配邏輯完全不動**  
- 只在「一次派幾個」這裡插入動態上限

---

### **不需要改的東西** {#不需要改的東西}

| 項目 | 狀態 |
| :---- | :---- |
| Timeout 判斷邏輯（仍 2 分鐘） | 不動 |
| VIP 分配邏輯 | 不動 |
| 機器健康度評估 | 不動 |
| 現有接續回測功能 | 不動 |

---

### **預期效果** {#預期效果}

| 策略類型 | Probe 耗時 | safe\_concurrency | 結果 |
| :---- | :---- | :---- | :---- |
| 計算量輕 | 短（\< 10s） | 高（\>12） | 速度與現在相同 |
| 計算量重 | 長（\> 60s） | 低（1\~2） | 每批少送，timeout 率大幅下降 |

## **決策記錄** {#決策記錄}

| \# | 問題 | 決策 |
| :---- | :---- | :---- |
| 1 | Probe 商品選取 | **Server 自動選**，從整批清單中取成交量前 N 檔 |
| 2 | Probe 全部 timeout | **直接 fallback**，`safe_concurrency = 1` |
| 3 | 校準結果快取 | **需要**，同策略在有效期內重跑可省略 probe（快取 key 建議：策略 ID \+ 商品清單 hash，TTL 待 RD 評估） |

## **長期方向** {#長期方向}

等 RD 有餘裕時，實作真正的**動態 timeout 偵測**：偵測 K 棒是否在推進中，若推進則自動延長 timeout，真正卡死才算 timeout。這是從根本解決問題的方向。

## **根本解法：Per-K-bar Timeout（取代整體 timeout）**

### **問題重新定義**

固定整體 timeout（2 分鐘/商品）本質上不合理：

- 一支 5000 根 K 棒、每根 1 秒的股票（總計 83 分鐘）是合理的，不該被 kill  
- 真正需要 kill 的只有「卡在同一根 K 棒不推進」的無窮迴圈

### **演算法**

```
1. 先跑前 N 根 K 棒（例如 10 根）暖機，收集 per-bar 耗時
2. avg_bar_time = 前 N 根的平均耗時
3. dynamic_threshold = avg_bar_time × 2
4. effective_threshold = min(dynamic_threshold, admin_hard_limit)
5. 後續每根 K 棒：距上次推進超過 effective_threshold → kill 該商品
```

### **RD 需實作（最小修改）**

| 項目 | 說明 |
| :---- | :---- |
| K 棒推進 hook | 運算引擎每推進一根 K 棒，記錄 timestamp |
| Watchdog | 排程器監控 per-bar 間隔，超過 `effective_threshold` 才 kill |
| Admin 設定 | 系統管理員設定 `admin_hard_limit`（全域上限，防止異常策略佔用過久） |

### **現有整體 timeout 的處置**

可逐步退場：初期兩者並存（per-bar timeout 先上，整體 timeout 調大作為保險），待觀察穩定後移除整體 timeout。
